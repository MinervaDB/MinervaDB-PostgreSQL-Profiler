## Overview

MinervaDB PostgreSQL Profiler is a kernel-level observability tool that instruments PostgreSQL without modifying the database binary or requiring restarts. It combines five eBPF attachment mechanisms to capture every query execution, lock acquisition, buffer access, WAL write, and scheduler event at nanosecond resolution.

---

## eBPF Program Design

Each profiling module is an independent eBPF program compiled to BPF bytecode via clang/LLVM and loaded at runtime using libbpf. Programs are structured around the **BPF skeleton** pattern generated by `bpftool gen skeleton`.

### Module to File Mapping

| Module | Source File | Program Type | Verifier Budget |
|---|---|---|---|
| Query Profiler | `ebpf/query_profiler.bpf.c` | BPF_PROG_TYPE_TRACING (USDT) | ~500K insns |
| Lock Profiler | `ebpf/lock_profiler.bpf.c` | BPF_PROG_TYPE_KPROBE | ~300K insns |
| I/O Profiler | `ebpf/io_profiler.bpf.c` | BPF_PROG_TYPE_TRACEPOINT | ~400K insns |
| Memory Profiler | `ebpf/memory_profiler.bpf.c` | BPF_PROG_TYPE_TRACING (uprobe) | ~600K insns |
| WAL Profiler | `ebpf/wal_profiler.bpf.c` | BPF_PROG_TYPE_TRACING (uprobe+USDT) | ~350K insns |
| CPU Profiler | `ebpf/cpu_profiler.bpf.c` | BPF_PROG_TYPE_PERF_EVENT | ~200K insns |
| Wait Profiler | `ebpf/wait_profiler.bpf.c` | BPF_PROG_TYPE_TRACING (USDT) | ~250K insns |
| Vacuum Profiler | `ebpf/vacuum_profiler.bpf.c` | BPF_PROG_TYPE_TRACING (USDT) | ~200K insns |
| Connection Profiler | `ebpf/conn_profiler.bpf.c` | BPF_PROG_TYPE_KPROBE | ~300K insns |
| Replication Profiler | `ebpf/repl_profiler.bpf.c` | BPF_PROG_TYPE_TRACING (uprobe) | ~350K insns |

### Program Lifecycle

```
Load phase (userspace):
  1. open_and_load()       -- clang compiles .bpf.c -> .o -> libbpf loads into kernel
  2. skeleton__attach()    -- programs attached to probe points
  3. ring_buffer__new()    -- poll loop begins

Teardown (on SIGINT / timeout):
  4. ring_buffer__free()
  5. skeleton__destroy()   -- programs detached, maps freed
```

## Attachment Points

### USDT Probes (User Statically Defined Tracing)

USDT probes are the preferred attachment mechanism. PostgreSQL must be compiled with `--enable-dtrace`. Probe sites are embedded as NOP instructions in the binary and activated at near-zero cost.

Key USDT probes used:

| Probe | PostgreSQL Source | Purpose |
|---|---|---|
| `postgresql:query__start` | `src/backend/tcop/postgres.c` | Query begin timestamp |
| `postgresql:query__done` | `src/backend/tcop/postgres.c` | Query end, compute latency |
| `postgresql:query__parse__start` | `src/backend/parser/parser.c` | Parse phase start |
| `postgresql:query__execute__start` | `src/backend/executor/execMain.c` | Execute phase start |
| `postgresql:lock__wait__start` | `src/backend/storage/lmgr/lock.c` | Lock wait begin |
| `postgresql:lock__wait__done` | `src/backend/storage/lmgr/lock.c` | Lock wait end |
| `postgresql:deadlock__found` | `src/backend/storage/lmgr/deadlock.c` | Deadlock detection |
| `postgresql:wal__insert` | `src/backend/access/transam/xlog.c` | WAL record written |
| `postgresql:checkpoint__start` | `src/backend/postmaster/checkpointer.c` | Checkpoint start |
| `postgresql:autovacuum__start` | `src/backend/postmaster/autovacuum.c` | Autovacuum run |

### kprobes and kretprobes

Used where USDT probes are unavailable or where kernel-level context is needed (e.g., TCP connection tracking for the Connection Profiler).

```c
// Connection profiler attaches to tcp_v4_connect
SEC("kprobe/tcp_v4_connect")
int BPF_KPROBE(trace_tcp_connect, struct sock *sk) { ... }
```

### uprobes and uretprobes

Used for `palloc` / `palloc0` (Memory Profiler) and WAL sender/receiver functions that lack USDT probes in older PostgreSQL builds.

```c
// Memory profiler: uprobe on palloc
SEC("uprobe/palloc")
int BPF_UPROBE(trace_palloc, Size size) { ... }
```

### tracepoints

Used in the I/O profiler for block layer events which are stable across kernel versions.

```c
SEC("tracepoint/block/block_rq_complete")
int trace_block_rq_complete(struct trace_event_raw_block_rq_complete *ctx) { ... }
```

### perf_events (CPU Profiler)

The CPU profiler uses `perf_event_open` with `PERF_TYPE_SOFTWARE / PERF_COUNT_SW_CPU_CLOCK` at 99 Hz to collect stack traces from all PostgreSQL backend processes.

## Data Flow

```
Kernel space:
  PostgreSQL backend process
       |
       v (USDT/uprobe/kprobe/tracepoint fires)
  eBPF program executes (< 1 us)
       |
       |-- reads context registers (PT_REGS)
       |-- reads BPF map (per-query state, timing data)
       |-- computes delta (e.g., query_end_ns - query_start_ns)
       +-- submits event struct to ring buffer

Userspace:
  ring_buffer__poll()  <- epoll wait, wakes on new events
       |
       v
  handle_event() callback
       |
       |-- aggregates into in-memory hash table
       |-- updates Prometheus gauges/histograms
       +-- writes to JSON/CSV output stream
```

**Event structs** are defined in `ebpf/common.h` and shared between kernel and userspace via BTF CO-RE:

```c
struct query_event {
    __u64  start_ns;
    __u64  end_ns;
    __u32  pid;
    __u32  db_oid;
    char   query[256];       // truncated to 256 bytes
    char   dbname[64];
    __u8   had_error;
    __u8   pad[7];
};
```

## BPF Map Layout

| Map Name | Type | Key | Value | Max Entries | Purpose |
|---|---|---|---|---|---|
| `query_start_map` | BPF_MAP_TYPE_HASH | u32 pid | u64 start_ns | 65536 | Track query start time per PID |
| `query_stats_map` | BPF_MAP_TYPE_HASH | u64 query_hash | struct query_stats | 65536 | Aggregate per-query stats |
| `lock_wait_map` | BPF_MAP_TYPE_HASH | u32 pid | struct lock_state | 65536 | Per-PID in-progress lock wait |
| `io_start_map` | BPF_MAP_TYPE_HASH | u64 sector | u64 start_ns | 65536 | Block I/O start time |
| `stack_traces` | BPF_MAP_TYPE_STACK_TRACE | u32 stack_id | u64[127] | 8192 | CPU flame graph stacks |
| `cpu_counts` | BPF_MAP_TYPE_HASH | u64 stack_id | u64 count | 65536 | Stack sample counts |
| `events` | BPF_MAP_TYPE_RINGBUF | — | union event_data | 64 MB | Kernel to userspace event channel |
| `config_map` | BPF_MAP_TYPE_ARRAY | u32 idx | u64 value | 16 | Runtime config (thresholds etc.) |

Map sizes are configurable via `config/profiler.yaml` under `ebpf.map_max_entries`.

## Ring Buffer Architecture

The profiler uses `BPF_MAP_TYPE_RINGBUF` (introduced in Linux 5.8) instead of the older `BPF_MAP_TYPE_PERF_EVENT_ARRAY` for several reasons:

- **Ordering:** Events are delivered in kernel-timestamp order across CPUs.
- **No per-CPU buffers:** Single shared buffer avoids cross-CPU aggregation complexity.
- **Backpressure:** If userspace falls behind, the kernel drops events and increments a counter rather than blocking.

On kernels earlier than 5.8, the collector automatically falls back to `BPF_MAP_TYPE_PERF_EVENT_ARRAY` with per-CPU buffers of the same aggregate size.

Monitor drops:

```bash
# Via Prometheus
curl -s localhost:9187/metrics | grep profiler_ringbuf_drops
```

## BTF CO-RE Portability

All eBPF programs use **BPF CO-RE (Compile Once - Run Everywhere)** via BTF (BPF Type Format). This eliminates the need to ship kernel headers or recompile programs per kernel version.

Requirements:

- Kernel must have `CONFIG_DEBUG_INFO_BTF=y` (standard in Ubuntu 20.04+, RHEL 9+, Amazon Linux 2023)
- `/sys/kernel/btf/vmlinux` must be readable

Verify:

```bash
ls -la /sys/kernel/btf/vmlinux
bpftool btf dump file /sys/kernel/btf/vmlinux format c > /dev/null && echo "BTF OK"
```

## Userspace Collector

The userspace collector lives in `collector/` and is written in Python 3.

```
collector/
|-- profiler.py      # Main entry point, CLI argument parsing
|-- loader.py        # eBPF program loader (libbpf skeleton or BCC)
|-- aggregator.py    # In-memory stats aggregation, percentile computation
|-- prometheus.py    # Prometheus HTTP server, gauge/histogram registration
|-- flamegraph.py    # Stack trace folding -> SVG flame graph generation
|-- output.py        # JSON / CSV / text formatter
+-- config.py        # Configuration file parser (YAML)
```

Percentiles (p50, p95, p99) are computed using the t-digest algorithm for constant-memory streaming approximation.

```
eBPF ring buffer
```

## Security Model

| Capability | Required For | Minimum Kernel |
|---|---|---|
| `CAP_BPF` | Loading eBPF programs | 5.8 |
| `CAP_PERFMON` | perf_events, reading kernel symbols | 5.8 |
| `CAP_SYS_PTRACE` | Reading /proc/<pid>/mem for uprobe symbol resolution | All |
| `root` | Full fallback on kernels older than 5.8 | < 5.8 |

On Linux 5.8+, the profiler can run as a non-root user with capabilities granted via file capabilities:

```bash
sudo setcap cap_bpf,cap_perfmon,cap_sys_ptrace+eip /usr/local/bin/minervadb-profiler
```

## Performance Design

**Zero-copy event submission** — The eBPF ring buffer uses `bpf_ringbuf_reserve()` and `bpf_ringbuf_submit()` to write directly into a shared memory region mapped into userspace, eliminating kernel-to-userspace copy overhead.

**Per-CPU aggregation in kernel** — Where possible (lock wait counters), BPF per-CPU arrays are used to avoid spinlock contention on multi-core systems.

**Adaptive sampling** — The query profiler skips recording queries below the configured `query_slow_threshold_ms` entirely in the eBPF program, so fast queries never reach the ring buffer.

**Memory budget** — Each BPF map pre-allocates memory at load time. Total kernel memory usage with all modules enabled and default map sizes: approximately 250 MB. Adjust `map_max_entries` in config if memory is constrained.

See [docs/tuning-guide.md](tuning-guide.md) for sizing recommendations and preset configurations.
